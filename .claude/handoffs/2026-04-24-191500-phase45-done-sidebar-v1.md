## Key Patterns Discovered (Zellij Plugin Dev)

1. **Binary crate, not cdylib** — `register_plugin!` generates `main()` and `_start`. Without `_start` export, plugin loads but never renders.
2. **Artifact naming** — binary crate produces `rally-plugin.wasm` (hyphen), cdylib produces `rally_plugin.wasm` (underscore). Easy to copy the stale one.
3. **Plugin cache** — Zellij caches wasm at `~/Library/Caches/org.Zellij-Contributors.Zellij/`. Use `skip_plugin_cache true` in layout KDL during dev.
4. **Permissions** — `request_permission()` shows an interactive dialog inside the plugin pane. On narrow panes, the dialog is invisible and blocks all rendering. Pre-grant via `permissions.kdl` with the **absolute path** as the key (no `file:` prefix).
5. **`run_command` is async** — returns results via `Event::RunCommandResult`. Use `BTreeMap<String,String>` context to tag which command produced which result.
6. **Rendering** — `println!` to stdout, zellij captures it via `wasi_read_string`. Only called when `update()` returns `true`.

## Decisions Made

| Decision | Rationale |
|----------|-----------|
| `run_command` over `zellij pipe` for v1 | Simpler, no daemon changes needed. Pipe is a performance optimization for Phase 8 (50 agents). |
| Skip `request_permission` dialog, pre-grant | Dialog blocks render in narrow sidebar pane. No way to auto-grant in layout KDL. |
| Remove Phase 6→7 dependency | Sidebar reads from daemon via CLI, not MCP. MCP is for agent-to-agent queries. |
| Exclude rally-plugin from default-members | Can't build wasm target with `cargo build --workspace`. |
